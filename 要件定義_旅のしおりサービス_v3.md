## 1. プロジェクト概要

### 1.1 サービスコンセプト

AIとの対話を通じて、旅の計画から「しおり」の作成・共有までをワンストップで完結する
国内旅行向けWebサービス。SNS認証（Google / LINE）でワンクリックログインし、
家族やグループでの旅行計画を楽しく・スムーズにする。

### 1.2 サービス名候補

| # | 名前 | 読み | コンセプト |
|---|------|------|-----------|
| 1 | **しおりっぷ** | Shiorippu | 「しおり」+「trip」。かわいさと旅感が両立。覚えやすい |
| 2 | **たびしるべ** | Tabishirube | 「旅の道しるべ」。AIがガイドするイメージに合う。和の響き |
| 3 | **Tabily** | タビリー | 「tabi」+「easily/daily」。シンプルで国際的。ドメイン取りやすそう |
| 4 | **Voyori** | ヴォヨリ | 「voyage」+「しおり」。洗練された響き。ロゴ映えしそう |
| 5 | **たびのわ** | Tabinowa | 「旅の輪（仲間）」+「旅の和（調和）」。グループ旅行のコンセプトに合う |
| 6 | **Planiori** | プラニオリ | 「plan」+「しおり」。計画→しおりの機能が名前で伝わる |

> 🎯 個人的な推し: **しおりっぷ** — 日本語圏の家族・グループ旅行がターゲットなので、
> 親しみやすさとサービス内容が一発で伝わる名前が強い。
> ドメイン: `shiorippu.com` or `shiori-trip.com`

### 1.3 ターゲットユーザー

| 属性 | 詳細 |
|------|------|
| **メイン** | 家族旅行・友人グループ旅行の幹事 |
| **サブ** | カップル、少人数旅行 |
| **年齢層** | 20〜50代（スマホ・PC両方利用） |

### 2.2 認証設計（Supabase Auth）

```
■ 認証プロバイダー
  1. Google OAuth 2.0
     - Googleアカウントでワンクリックログイン
     - Supabase Auth ダッシュボードで設定
  2. LINE Login
     - LINE公式アカウント + LINE Login チャネルが必要
     - LINEユーザーが多い日本市場に最適
     - Supabase Auth の OIDC Custom Provider で設定

■ 認証フロー
  1. トップページ → 「Googleでログイン」or「LINEでログイン」
  2. OAuth画面 → 認証完了 → コールバック
  3. Supabase Auth がセッション管理
  4. Nuxt側は useSupabaseUser() / useSupabaseClient() で状態管理

■ Nuxt UI AuthForm コンポーネント活用
  - Nuxt UI に AuthForm が組み込み済み
  - providers prop で Google / LINE を設定するだけ

■ 認証ミドルウェア
  - /plan, /shiori/*/edit 等の編集ページは認証必須
  - /shiori/{id} (閲覧) は認証不要（共有リンクでアクセス可能）
```

### 2.3 データ永続化（Supabase一本）

```
■ 方針
  - 全データをSupabase PostgreSQLに保存
  - クライアント側はPiniaでキャッシュ（永続化はしない）
  - RLS (Row Level Security) でユーザー間のデータ分離

■ メリット
  - 複数デバイスからアクセス可能（PC→スマホ等）
  - データ消失リスクなし
  - 共同編集がシンプルに（同一データソース）
  - バックアップはSupabase側で自動

■ Pinia の役割（キャッシュのみ）
  - APIレスポンスのクライアントキャッシュ
  - 編集中の楽観的更新（Optimistic UI）
  - 未保存の変更の一時保持
  - 永続化プラグインは使わない
```

### 2.5 共同編集アーキテクチャ（Google Docs型）

```
■ リアルタイム同期
  Supabase Realtime（Presence + Broadcast + Postgres Changes）

  1. Postgres Changes: DBの変更をリアルタイム配信
     → 予定の追加/削除/更新を全クライアントに反映
  2. Presence: 接続中ユーザー情報
     → アバター表示、編集中フィールドのハイライト
  3. Broadcast: 軽量イベント配信
     → カーソル位置、ドラッグ中の状態など一時的情報

■ カーソル表示
  - 各編集者に色を割り当て（AvatarGroup で表示）
  - 編集中のカード/フィールドを色付きボーダーでハイライト
  - ニックネーム（Supabase Authのdisplay_name）表示

■ 競合解決
  - Supabase PostgreSQLの updated_at による楽観的ロック
  - 同一フィールド同時編集: Last Write Wins + Toast通知
  - 将来的にYjs等のCRDT導入を検討

■ 共有・権限モデル（招待リンク制）

  1. 閲覧共有
     - 共有リンクを知っていれば誰でも閲覧可能（認証不要）
     - LINE/SNSで手軽に共有できる

  2. 編集招待（招待リンク制）
     - オーナーが「招待リンク」を発行
     - 招待リンクを受け取った人がGoogleログインで参加
     - ログイン済みユーザーがcollaboratorsテーブルに登録される
     - 招待リンクは使い回し可（同じリンクで複数人が参加可能）
     - オーナーがリンクを無効化・再発行可能

  3. 権限レベル
     - 閲覧者: 共有リンクでアクセス（認証不要、編集不可）
     - 編集者: 招待リンク経由 + Googleログイン（編集可）
     - オーナー: 作成者（全権限: 削除、招待管理、設定変更）
```

## 4. データモデル

### 4.1 Supabase テーブル設計

```sql
-- ==========================================
-- ユーザー（Supabase Auth が自動管理）
-- auth.users テーブルに保存される
-- ==========================================

-- ==========================================
-- プロフィール（auth.users の拡張情報）
-- ==========================================
CREATE TABLE public.profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name TEXT,
  avatar_url TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- ==========================================
-- しおり（メインエンティティ）
-- ==========================================
CREATE TABLE public.shioris (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  owner_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title TEXT NOT NULL DEFAULT '新しいしおり',
  start_date DATE,
  end_date DATE,
  area TEXT,                              -- 旅行エリア
  template_id TEXT DEFAULT 'simple',      -- デザインテンプレート
  custom_style JSONB DEFAULT '{}',        -- カスタムスタイル
  cover_image_url TEXT,
  is_public BOOLEAN DEFAULT FALSE,        -- 閲覧用共有リンク有効
  invite_token TEXT UNIQUE,               -- 編集招待トークン（UUID）
  invite_enabled BOOLEAN DEFAULT FALSE,   -- 招待リンク有効/無効
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- ==========================================
-- 日程
-- ==========================================
CREATE TABLE public.days (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  shiori_id UUID NOT NULL REFERENCES public.shioris(id) ON DELETE CASCADE,
  day_number INT NOT NULL,
  date DATE,
  sort_order INT NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- ==========================================
-- 予定（イベント）
-- ==========================================
CREATE TABLE public.events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  day_id UUID NOT NULL REFERENCES public.days(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  category TEXT NOT NULL DEFAULT 'other',    -- EventCategory
  icon TEXT DEFAULT 'map-pin',               -- Lucide icon name
  start_time TIME,
  end_time TIME,
  memo TEXT,
  url TEXT,                                  -- 外部リンク
  image_url TEXT,
  place_id TEXT,                             -- Google Places ID
  lat DECIMAL(10, 7),
  lng DECIMAL(10, 7),
  address TEXT,
  booking_status TEXT DEFAULT 'none',        -- none, pending, confirmed, cancelled
  sort_order INT NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- ==========================================
-- AI会話履歴
-- ==========================================
CREATE TABLE public.chat_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  shiori_id UUID NOT NULL REFERENCES public.shioris(id) ON DELETE CASCADE,
  role TEXT NOT NULL,                        -- 'user' | 'assistant'
  content TEXT NOT NULL,
  metadata JSONB DEFAULT '{}',               -- 提案データ等
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- ==========================================
-- 共同編集者
-- ==========================================
CREATE TABLE public.collaborators (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  shiori_id UUID NOT NULL REFERENCES public.shioris(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role TEXT NOT NULL DEFAULT 'editor',       -- 'owner' | 'editor'
  last_active TIMESTAMPTZ DEFAULT NOW(),
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(shiori_id, user_id)
);

-- 招待リンク経由で参加するフロー:
-- 1. オーナーが invite_token を生成（shioris.invite_enabled = true）
-- 2. /shiori/{id}/invite/{invite_token} のURLを共有
-- 3. 受け取った人がアクセス → 未ログインならGoogleログインへ
-- 4. ログイン後、collaboratorsテーブルにINSERT（role='editor'）
-- 5. 以降は通常の編集者として参加

-- ==========================================
-- RLS ポリシー
-- ==========================================

-- profiles: 自分のプロフィールのみ編集可、全員閲覧可
ALTER TABLE public.profiles ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Profiles are viewable by everyone" ON public.profiles FOR SELECT USING (true);
CREATE POLICY "Users can update own profile" ON public.profiles FOR UPDATE USING (auth.uid() = id);

-- shioris: オーナーが全操作可、公開しおりは誰でも閲覧可
ALTER TABLE public.shioris ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Owner can do everything" ON public.shioris FOR ALL USING (auth.uid() = owner_id);
CREATE POLICY "Public shioris are viewable" ON public.shioris FOR SELECT USING (is_public = true);

-- days / events: しおりのオーナーまたは共同編集者が操作可
ALTER TABLE public.days ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.events ENABLE ROW LEVEL SECURITY;
-- (詳細ポリシーは実装時に設計)

-- インデックス
CREATE INDEX idx_shioris_owner ON public.shioris(owner_id);
CREATE INDEX idx_days_shiori ON public.days(shiori_id);
CREATE INDEX idx_events_day ON public.events(day_id);
CREATE INDEX idx_chat_messages_shiori ON public.chat_messages(shiori_id);
CREATE INDEX idx_collaborators_shiori ON public.collaborators(shiori_id);
```

## 8. AI プロンプト設計

### 8.1 AI応答形式 — 選択肢型 + 自由入力

```
■ 基本ルール
  - AIは各ヒアリング項目を「選択肢カード」形式で提示する
  - 各質問には3〜4個の選択肢 + 「その他（自由入力）」を含む
  - ユーザーはタップ/クリックで回答 or 「その他」でテキスト入力
  - 選択後、AIが次の質問に進む（ストリーミングで応答）
  - スマホでもタップだけで進められるUXを重視

■ AIの応答JSON構造
  AIは通常テキスト + 選択肢データのハイブリッドで応答する。
  選択肢は present_choices ツールで構造化して返す。

■ ヒアリングフロー（選択肢の具体例）

  Q1: 行き先
  テキスト: 「どちらに行きたいですか？✈️」
  選択肢:
    - 🌸 北海道（大自然・グルメ）
    - ⛩️ 京都・奈良（歴史・文化）
    - 🌋 九州（温泉・自然）
    - 🏖️ 沖縄（ビーチ・リゾート）
    - ✏️ その他（自由入力）

  Q2: 日程
  テキスト: 「何泊何日の旅行ですか？📅」
  選択肢:
    - 🌙 1泊2日（週末旅行）
    - 🌙🌙 2泊3日（定番）
    - 🌙🌙🌙 3泊4日以上（じっくり）
    - ✏️ その他（自由入力）

  Q3: 人数・構成
  テキスト: 「誰と行きますか？👨‍👩‍👧‍👦」
  選択肢:
    - 👫 カップル・夫婦（2人）
    - 👨‍👩‍👧‍👦 家族（子供あり）
    - 👯 友人グループ（3人以上）
    - ✏️ その他（自由入力）
  ※「家族」選択時 → 追加質問「お子さんの年齢を教えてください」

  Q4: 旅のテーマ（複数選択可）
  テキスト: 「どんな旅にしたいですか？🎯（複数選択OK）」
  選択肢:
    - 🍜 グルメ・食べ歩き
    - ♨️ 温泉でリラックス
    - 🏯 観光名所めぐり
    - 🎡 子供が楽しめる体験
    - 🏔️ 自然・アウトドア
    - ✏️ その他（自由入力）

  Q5: 予算
  テキスト: 「予算はどのくらいですか？💰」
  選択肢:
    - 💴 〜5万円（リーズナブル）
    - 💴💴 5〜10万円（スタンダード）
    - 💴💴💴 10〜20万円（ちょっと贅沢）
    - 💴💴💴💴 20万円以上（プレミアム）
    - ✏️ その他（自由入力）

  Q6: 出発地・交通手段
  テキスト: 「どこから出発しますか？🚗」
  選択肢:
    - 🚅 東京駅（新幹線）
    - ✈️ 羽田/成田空港（飛行機）
    - 🚗 車で出発（マイカー/レンタカー）
    - ✏️ その他（自由入力）

  Q7: 特記事項
  テキスト: 「他に気をつけることはありますか？📝」
  選択肢:
    - 🍽️ アレルギーあり
    - ♿ バリアフリー希望
    - 🐕 ペット同伴
    - ✅ 特になし
    - ✏️ その他（自由入力）
```

### 8.2 システムプロンプト

```
あなたは国内旅行のプロフェッショナルプランナーです。
ユーザーの要望を丁寧にヒアリングし、具体的で実現可能な旅行プランを提案します。

## 応答ルール
- 各質問は必ず present_choices ツールで選択肢を提示する
- 選択肢は3〜5個 + 「その他（自由入力）」を含む
- 会話テキストは親しみやすく、絵文字を適度に使う
- ユーザーが「その他」で自由入力した場合は柔軟に対応
- 複数選択可の質問にはmultiSelect: trueを指定
- 各スポット提案には必ず search_spots / get_spot_details を呼ぶ
- プラン完成時は create_trip_plan で構造化データを出力
- 移動時間は calculate_route で正確に算出
- 予算は具体的な金額レンジで提示
```
